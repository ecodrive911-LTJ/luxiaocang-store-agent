# 鹿小仓 · 工作流硬约束（MANDATORY）

> **本文件是强制约束，优先级最高。**
> 任何对鹿小仓代码 / 配置 / 数据结构的改动，必须严格按以下流程执行，不得跳过任何一步。
> Agent（包括人类工程师与 AI 助手）在执行前必须主动宣读本文件并逐条打勾。

---

## 一、每次代码改动必走链路（5 步，缺一不结）

| 步骤 | 动作 | 验收标准 |
|------|------|----------|
| 1️⃣ 本地测试 | 语法检查 + 关键场景验证 | `python -m py_compile` 通过；相关场景本地复现通过 |
| 2️⃣ GitHub 备份 | `git add` → `commit` → `push origin master:main` | 远程仓库可见最新 commit |
| 3️⃣ 云端部署 | `python upload_to_server.py`（SFTP 到 `/opt/luxiaocang/` + `systemctl restart luxiaocang.service`） | 服务 `active (running)`，版本号正确 |
| 4️⃣ 线上验证 | 登录真实账号，发关键场景消息或调关键接口 | 功能与本地一致，无报错 |
| 5️⃣ 变更记录 | 在 `TASKBOARD.md` 变更记录 + 当日 `memory/YYYY-MM-DD.md` 双写 | 两处均有记录，内容一致 |

---

## 二、反例（禁止出现）

以下任一情形均视为**违规**，必须返工：

- ❌ 只在本地改完就说"搞定了"
- ❌ 上传了但没 GitHub 备份
- ❌ GitHub 推了但没线上验证
- ❌ 线上验证了但没写变更日志
- ❌ 跳步（如"先部署再说，GitHub 后面补"）

## 四、特殊场景补充

### 4.1 紧急热修复
可先做步骤 3（部署）止血，但**必须在 24 小时内补齐步骤 2（GitHub）和步骤 5（变更日志）**，否则视为违规。

### 4.2 纯文档改动
仅修改 `*.md` 文件、不影响运行时行为的改动，可省略步骤 3 和 4，但仍需步骤 2 和 5。

### 4.3 数据库结构变更
必须先备份 `database.db`（本地 + 服务器各一份），再执行建表/改表操作。
