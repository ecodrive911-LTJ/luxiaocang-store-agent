## 一、总体架构：总部管控 + 门店执行

### 1.1 四端口定位

| 端口 | 使用者 | 角色 | 核心职能 |
|------|--------|------|---------|
| **总部完整版 Web** | 管理员(admin) / 总部管理(manager) | 决策 + 管控 | 全局数据监控、任务下发、系统配置、用户管理、跨店分析 |
| **门店微信小程序**（计划阶段二） | 分店店主(store_owner) / 店员 | 执行 + 咨询 | 接收任务、视频采集上传、AI轻量化咨询（仅本店） |
| **门店轻量化 Web**（备用） | 分店店主(store_owner) | 查看 + 执行 | 大屏查看本店报告、备用的任务执行入口 |
| **独立运维后台**（计划中） | 系统运维 | 技术管控 | 服务监控、故障排查、配置管理 |

### 1.2 总部-门店核心关系

```
总部做什么（决策与管控）
  ├─ 全局数据监控：看所有门店 + 所有竞品数据
  ├─ 任务下发：选定门店 → 选定品类/商品 → 设定截止时间 → 下发
  ├─ 结果分析：任务采集结果 → AI分析 → 人工复核 → 决策
  ├─ 系统配置：大模型、地图API、功能开关 → 全局生效
  └─ 用户管理：创建账号、分配角色、绑定门店

门店做什么（执行与反馈）
  ├─ 接收任务：看到总部下发给自己的采集任务
  ├─ 执行采集：按任务清单拍摄/录屏 → 上传
  ├─ AI轻咨询：问本店的运营诊断、调价建议、滞销提醒
  ├─ 查看结果：看到自己上传的解析结果
  └─ 本店管理：库存预警、临期提醒、本店利润

总部与门店的关联与反馈回路
  ┌────────────────────────────────────────────────────────┐
  │                                                         │
  │  总部创建任务 ──→ 门店接收 ──→ 门店采集上传            │
  │       ↑                          │                       │
  │       │                          ↓                       │
  │  决策行动 ←── AI分析报告 ←── 解析入库                   │
  │       │                          ↑                       │
  │       ↓                          │                       │
  │  下发新任务 ──→ 门店执行 ──→ 效果追踪 → 闭环            │
  │                                                         │
  └────────────────────────────────────────────────────────┘
```

### 1.3 关键原则

1. **数据归属**：门店数据归门店，总部可见全部；门店不可见其他门店数据，不可见全局竞品数据
2. **任务单向下发**：总部创建任务，门店执行；门店不能创建跨店任务
3. **AI 能力分层**：总部可做全局分析（跨店对比），门店只能做本店分析（本店诊断、本店调价）
4. **竞品数据隔离**：竞品数据由总部统一管理，门店只看到"自己被分配去采集哪些竞品"，不看到所有竞品数据库

## 二、十大模块逐项对标：现状 → 目标 → 行动计划

### 模块 0 · 数据采集

**总部职能**：下发采集任务、查看所有门店采集数据、人工复核AI解析结果、数据修正留痕  
**门店职能**：接收本店任务、上传视频/录屏、查看本店采集记录与解析结果

#### 现状
| 能力项 | 状态 | 说明 |
|--------|------|------|
| 创建任务 | ✅ 已实现 | POST `/api/tasks`，manager/admin 可创建 |
| 任务商品清单 | ✅ 已实现 | `collect_task_items` 表，支持逐商品下发 |
| 门店接收任务 | ✅ 后端已实现 | GET `/api/tasks` 按角色隔离（store_owner 只看到本店） |
| 视频上传 | ✅ 已实现 | POST `/api/tasks/{id}/upload`，50MB限制 |
| AI视频解析 | ⚠️ 部分实现 | Qwen视觉模型已在服务器部署，但未对接到任务系统自动触发 |
| 解析结果入库 | ⚠️ 部分实现 | `price_data` 表已建，但缺少自动写入+回调逻辑 |
| 人工复核 | ❌ 未实现 | 无人工审核界面、无修正留痕 |
| 失败退回重传 | ❌ 未实现 | 解析失败只记录错误，无通知门店重传机制 |
| 前端校验 | ❌ 未实现 | 无格式/大小/时长/MD5前端校验 |
| 总部查看全局采集 | ⚠️ 部分实现 | API有数据隔离，但前端无"全部门店采集数据"面板 |

#### 目标状态
- 总部：创建任务（强制绑定商品清单） → 下发 → 门店上传 → AI自动解析 → 解析结果通知总部 → 总部人工复核（修正留痕） → 数据入库
- 门店：接收任务 → 手机拍摄/录屏（前端校验格式/大小/时长/MD5） → 上传 → 看到解析状态 → 失败则看到原因并重传

#### 行动计划（第一阶段 7-14天）
1. **前端视频校验**（2天）：格式MP4、大小≤200M、时长10s-10min、MD5去重
2. **后端上传校验强化**（1天）：任务归属校验（跨店拦截）、重复上传拦截、任务状态校验（超时禁止上传）
3. **AI解析自动触发**（2天）：上传成功后 → 后台异步调用Qwen → 解析成功写price_data → 更新upload状态
4. **解析失败退回机制**（1天）：解析失败 → 更新upload状态+error_message → 门店端显示重传按钮
5. **人工复核界面**（2天）：总部Web端 → 看到低置信度(<0.7)的解析记录 → 人工修正 → 修正日志留存
6. **双向通知**（1天）：解析完成/失败 → 总部看到汇总 → 门店看到自己那条的结果

## 三、前端差异化改造（最高优先级，贯穿所有模块）

### 3.1 现状问题

当前三种角色登录后看到的是**几乎相同的界面**，只有三个按钮做了差异化：
```javascript
admin    → 任务✅ 设置✅ 用户✅
manager  → 任务✅ 设置✅ 用户❌
store_owner → 任务✅ 设置❌ 用户❌
```

左侧面板、数据资产、能力模块、快捷入口卡片——全部一样。

### 3.2 改造目标

| 区域 | admin/manager（总部） | store_owner（门店） |
|------|----------------------|---------------------|
| **顶部工具条** | 📋任务 ⚙️设置 👥用户 🚪登出 | 📋任务（仅自己） 🚪登出 |
| **当前用户** | 姓名 + 角色标签 | 姓名 + 本店名称 |
| **数据资产** | 全部门店+竞品数据（13,982条） | 仅本店数据（如广安店2,941条） |
| **能力模块** | 全部10个模块 | 隐藏：选址评估、建店规划、品牌管理（仅显示7个） |
| **系统状态** | 大模型+采集引擎+高德API | 隐藏（不需要了解底层） |
| **快捷入口卡片** | 全部6个 | 隐藏：商圈扫描、门店对比（仅显示4个：运营诊断、竞品采集、选品规划、定价策略） |
| **门店选择器** | 可切换所有门店 | 固定本店（不可切换）或只显示本店 |
| **AI对话上下文** | 可询问任意门店+竞品 | 自动绑定本店，不可越权 |

### 3.3 行动计划（第一阶段 4-10天）

1. **RBAC前端渲染逻辑**（2天）：`afterLogin()` 中根据 role 动态显示/隐藏左侧面板各区块
2. **数据资产隔离**（1天）：store_owner 的 `/api/data/assets` 只返回本店数据
3. **能力模块过滤**（1天）：store_owner 自动隐藏隐藏模块
4. **快捷卡片过滤**（1天）：store_owner 去掉"商圈扫描""门店对比"
5. **AI对话权限**（2天）：后端 `/api/chat` 中 store_owner 自动绑定本店，对话中不返回其他门店数据
6. **自测与修复**（3天）：三个角色分别登录验证
