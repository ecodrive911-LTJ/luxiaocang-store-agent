### 二、流程缺失类（断头路，无闭环）

| 编号 | 缺陷 | 代码证据 | 严重度 |
|------|------|---------|--------|
| P-01 | 视频采集流程断裂 | 上传 API (`POST /api/tasks/{id}/upload`) → 存文件到 `uploads/` → 写数据库 `collect_uploads` → **结束**。不触发分析、不回调、不通知 | 🔴 阻塞 |
| P-02 | 任务下发→执行→反馈无回路 | 总部创建任务 → 门店看到 → 门店上传 → **总部不知道上传了什么、解析了什么** | 🔴 阻塞 |
| P-03 | AI对话无上下文持久化 | 对话历史存 `conversations` 表，但 Agent 无记忆：每次对话独立无关联，不会记住"上次我让你分析广安店，现在继续" | 🟡 中等 |
| P-04 | 无调价→执行→追踪闭环 | 如果能生成调价方案，无下发机制、无执行确认、无效果追踪 | 🔴 阻塞 |
| P-05 | 无异常→通知→处理闭环 | 脚本报错、解析失败、超时——只记录 error_message，没有任何通知机制 | 🟡 中等 |

### 三、AI心智缺失类（被动工具，非智能体）

| 编号 | 缺陷 | 代码证据 | 严重度 |
|------|------|---------|--------|
| A-01 | 纯粹被动问答 | `agent_loop.py` 只响应用户输入，不会主动发现异常、主动推送预警、不会说"广安店的可口可乐连续3天比竞品贵，要不要调价？" | 🔴 阻塞 |
| A-02 | 无规划与自主执行能力 | Agent 每次最多5轮工具调用（`max_iterations=5`），无长期目标规划，无法分解复杂任务为多步骤自动执行 | 🔴 阻塞 |
| A-03 | 无记忆与学习能力 | 每次对话独立，不积累"用户偏好/门店特征/历史决策"。无长期记忆（`MEMORY.md` 尚未整合进 Agent loop） | 🟡 中等 |
| A-04 | 工具调用浅层 | 仅5个注册工具（`get_time/list_stores/list_competitors/run_script/search_web`），无数据库查询工具、无数据分析工具、无决策生成工具 | 🟡 中等 |
| A-05 | 无多轮推理链 | LLM返回工具调用→执行→返回结果→LLM汇总→结束。无"分析→发现问题→深入查询→再分析→出方案"的推理链 | 🟡 中等 |

### 五、安全与运维缺失类 [v2.1新增]

| 编号 | 缺陷 | 代码证据 | 严重度 |
|------|------|---------|--------|
| S-01 | **零操作审计** | 无任何操作日志表，总部管理员修改调价、删除数据、变更权限、数据导出均无留痕。一旦出现误操作或恶意操作，无法追溯责任人和时间 | 🔴 阻塞 |
| S-02 | **敏感数据无脱敏** | 门店营收、供应商进价等商业敏感数据对 store_owner 无任何脱敏或隐藏，且数据导出无权限管控、无水印 | 🔴 阻塞 |
| S-03 | **文件上传无跨店访问控制** | 上传的视频文件存于服务器 `uploads/` 目录，若URL可被猜测或遍历，存在跨门店读取竞品采集视频的风险 | 🟡 中等 |
| S-04 | **无独立运维后台** | 服务监控、模型调度、API密钥管理、日志查询均需 SSH 到服务器手动操作，无任何可视化运维界面，故障排查效率极低 | 🟡 中等 |
| S-05 | **无版本管理与回滚机制** | 每次部署直接覆盖文件 + `systemctl restart`，无版本号记录、无灰度发布、无一键回滚。全量更新在多门店同时使用场景下风险极高 | 🟡 中等 |

## 模块2：四阶全周期进化落地总路线

```
阶段0（当前）        阶段1（2.5天）        阶段2（15-45天）        阶段3（45天+）
看得见·Demo          可用·流程闭环         好用·主动赋能          商用·自主决策
──────────────────────────────────────────────────────────────────────────────
被动问答             全流程自动化          AI主动服务            虚拟店长自主经营
演示能力             零人工兜底            降80%人工重复          成熟智能体
```

### 阶段0：现状（看得见·纯展示Demo）
- **定位**：有基础骨架，可以演示对话和基础功能
- **核心价值**：验证了多租户架构、任务采集系统、Agent Loop 技术可行性
- **核心缺陷**：无闭环、无差异化、无主动能力、6/10模块未落地、零安全审计
- **验收标准**：当前状态即基线，不做验收

### 阶段1：短期攻坚（可用·流程闭环 / 2.5天）
- **定位**：打通核心链路，消灭所有断头路，实现可上线使用的MVP
- **核心价值**：门店可独立使用，总部可管控全局，数据采集有始有终
- **整体落地目标**：①前端彻底分家 ②视频采集AI解析闭环 ③运营诊断可运行 ④定价算法上线 ⑤人工复核机制 ⑥全局操作审计日志 ⑦自动化回归测试
- **验收标准**：三角色分别登录 → 看到不同界面 → 总部下发任务 → 门店接收上传 → AI自动解析 → 总部复核入库 → 定价方案输出 → 审计日志记录所有关键操作 → 自动化回归全部通过（全链路可跑通）

### 阶段2：中期迭代（好用·主动赋能 / 15-45天）
- **定位**：AI从被动变为主动，从问答工具变为经营参谋
- **核心价值**：AI主动监控数据异常、主动推送预警、主动给建议，减少80%人工盯数据的工作
- **整体落地目标**：①微信小程序上线 ②AI主动巡检+异常推送 ③选品规划+商品分层 ④数据看板可视化 ⑤用户偏好学习 ⑥门店画像长期记忆存储 ⑦小程序端安全鉴权加固
- **验收标准**：门店店主每天打开小程序 → 看到AI推送的"今日异常提醒" → 一键查看诊断报告 → 无异常则"一切正常"；AI对话自动关联门店历史经营特征

### 阶段3：长期成熟（商用·自主决策 / 45天+）
- **定位**：AI从参谋变为准店长，具备自主经营决策能力
- **核心价值**：AI自动发现经营问题→自主制定方案→总部审批→自动执行→效果追踪→自动优化，形成完全自主的经营决策闭环
- **整体落地目标**：①自主经营决策引擎 ②供应链接管 ③财务自动核算 ④美团数据自动接入 ⑤多门店协同调度 ⑥独立运维后台上线 ⑦数据安全审计体系达标
- **验收标准**：AI在没有人工干预的情况下，自主完成一次从发现问题→制定方案→执行→追踪的完整经营闭环；运维后台可独立完成服务监控+故障定位+一键回滚

### 4.1 视频采集AI解析全闭环落地路线

**现状**：上传→存文件→写数据库→结束。video_pipeline.py（847行RapidOCR+帧分析）从未被调用。

**目标**：上传→校验→异步解析→入库→复核→修正→最终入库→溯源

**落地步骤（阶段1·Day 2-3）：**

```
步骤1：前端上传校验（Day2 D2-1）
├─ 格式：仅MP4/MOV，其他格式弹窗拦截
├─ 大小：≤200MB，超限前端FileReader预检+拦截
├─ 时长：读取video.duration，10s-10min，超出弹窗
├─ MD5：CryptoJS计算hash，先调API查重复→已存在则提示"已上传过"
└─ UI：上传进度条 + 校验结果实时反馈

步骤2：后端上传强化（Day2 D2-2）
├─ 任务归属：task.store_id必须∈user_stores
├─ 重复拦截：SELECT COUNT(*) FROM uploads WHERE task_item_id=? AND file_path包含同MD5
├─ 超时拦截：task.deadline < now() → "任务已截止"
└─ 状态联动：上传成功 → task_items.status='done' + task.status='in_progress'

步骤3：AI异步解析（Day2 D2-3）
├─ 解析流程适配 video_pipeline.py：
│   ├─ 抽帧（每秒1帧，video_pipeline已有FPS_INTERVAL=1）
│   ├─ RapidOCR识别（已有，HAS_OCR标识）
│   ├─ LLM语义聚合（已有）
│   └─ 输出：[{product_name, price, spec, promotion_text, confidence}]
├─ 调用方式：
│   ├─ 上传接口立即返回（不等待解析）
│   ├─ 后台 subprocess.run python3 scripts/video_pipeline.py <file_path> --task_id=X
│   ├─ 解析完成回调：读pipeline输出JSON → 写price_data → 更新upload状态
│   └─ 并发控制：同时最多3个解析任务（asyncio.Semaphore）
└─ 失败重试：最多2次，指数退避(30s/120s)

步骤4：解析失败退回（Day2 D2-4）
├─ upload.status: pending → processing → success/failed
├─ failed时写error_message（如"未检测到价格标签""画面模糊"等）
├─ 门店端轮询 GET /api/tasks/{id}/uploads → 显示状态图标🔄✅❌
└─ 失败记录旁显示"重新上传"按钮

步骤5：人工复核（Day3 D3-1）

└─ 可按修正人/时间追溯

最终入库：price_data(is_verified=1) → 成为比价矩阵和定价算法的可信数据源
```

#### [v2.1新增] 长期记忆数据表设计

**表1：store_profiles（门店经营画像）**
```sql
CREATE TABLE store_profiles (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    store_id INTEGER NOT NULL REFERENCES stores(id),
    profile_type TEXT NOT NULL,  -- 'business_feature'|'historical_issue'|'pricing_preference'|'category_strength'|'category_weakness'
    content_json TEXT NOT NULL,  -- {"summary":"饮料类SKU多但毛利率偏低","detail":{...},"evidence":["diagnosis_20260715","pricing_20260710"]}
    confidence REAL DEFAULT 0.5, -- 0-1，多次印证后升高
    source TEXT,                 -- 'diagnosis'|'pricing'|'user_feedback'|'manual'
    last_updated TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**表2：agent_memory（Agent长期记忆）**
```sql
CREATE TABLE agent_memory (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    store_id INTEGER NOT NULL REFERENCES stores(id),
    memory_type TEXT NOT NULL,   -- 'decision'|'event'|'preference'|'lesson'
    summary TEXT NOT NULL,       -- ≤200字记忆摘要
    detail_json TEXT,            -- 完整上下文
    source_conversation_id INTEGER REFERENCES conversations(id),
    importance INTEGER DEFAULT 1, -- 1-5，越高越长期保留
    last_recalled_at TIMESTAMP,
    recall_count INTEGER DEFAULT 0,
    archived INTEGER DEFAULT 0,  -- 1=超过30天且importance≤2的归档
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);
```

**召回逻辑：**
1. 对话开始时查 `store_profiles WHERE store_id=?` 全部画像（通常≤20条）
2. 查 `agent_memory WHERE store_id=? AND archived=0 ORDER BY importance DESC, created_at DESC LIMIT 10`
3. 拼入 system prompt：`[门店画像] 饮料类SKU多但毛利率偏低 | [历史决策] 2026-07-10 可口可乐调价至3.0元，销量环比+8% | [用户偏好] 常关注价格敏感品类`
4. 对话结束后，若触发了诊断/定价工具 → 调用LLM提取本次关键信息 → 写入store_profiles + agent_memory

#### [v2.1新增] 小程序安全鉴权与文件访问管控

**规则1：openid + store_id 双重鉴权**
```
小程序端每次API请求Header携带：
  Authorization: Bearer <session_token>
  X-Store-Id: <当前门店ID>

后端中间件校验：

  ③ store_owner角色 → 校验X-Store-Id是否∈user_stores

```

**规则2：会话安全机制**
```
├─ 小程序端会话超时：30分钟无操作自动登出，需重新wx.login

└─ 登录失败锁定：连续5次鉴权失败锁定账号15分钟
```

**规则3：视频文件访问权限控制**
```
├─ 上传文件存储路径规则：uploads/{store_id}/{task_id}/{timestamp}_{uuid}.mp4
│   → 禁止store_id平铺在可遍历的目录名中
├─ 文件访问API：GET /api/files/{file_id}
│   → 校验请求者store_id是否匹配文件的归属store_id
│   → 不匹配返回403，不暴露文件是否存在
├─ 禁止直接URL访问：Nginx配置uploads目录仅允许/api/files代理访问
│   location ^~ /uploads/ { deny all; }
└─ 文件URL加签名时效：生成临时访问URL时加HMAC签名+5分钟过期
```

### 6.3 迭代节奏优化建议

1. **阶段1绝不放过**：这是从0到1的质变，一次性解决所有断头路+安全审计基础
2. **阶段2优先启动小程序资质 + AI巡检**：审核周期不可控，越早提交越好；巡检不依赖小程序可先行
3. **阶段3中运维后台与审计加固可并行**：独立于AI决策引擎，可在阶段2末期提前开工
4. **版本管理与灰度发布机制建议阶段1即建立**：tag + 备份 + 回滚脚本成本极低（<0.5h），但价值巨大

### 6.4 [v2.1新增] 灰度发布标准化流程

```
每次部署标准流程：
1. 部署前：备份当前版本 → git tag v0.X.Y → git push --tags
2. 部署执行：SFTP上传文件 → systemctl restart luxiaocang → 等待3秒
3. 部署后验证：
   ├─ curl http://localhost:8000/api/health → 200
   ├─ python -m pytest tests/ -v → 全部PASS
   └─ 浏览器登录admin账号 → 功能冒烟测试
4. 验证失败回滚：
   ├─ git checkout <上一个tag> → 恢复文件
   ├─ systemctl restart luxiaocang
   └─ 再次验证 → 仍失败则告警通知

阶段3灰度发布增强：
├─ 灰度路径：/opt/luxiaocang-gray/ 独立运行在8001端口
├─ 灰度账号：仅ops和指定测试账号路由到8001
├─ Nginx规则：根据Cookie gray_user=1 路由到灰度后端
├─ 灰度验证通过 → 8001覆盖到8000 → 全量生效
└─ 灰度验证失败 → 删除灰度实例 → 不影响8000主服务
```
