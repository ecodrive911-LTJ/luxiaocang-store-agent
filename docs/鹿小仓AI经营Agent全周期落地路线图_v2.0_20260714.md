## 模块1：项目现状精准诊断

### 当前阶段定位

**🟡 阶段0：看得见 — Demo 展示阶段**

虽然代码已经迭代到 v0.5，但从核心闭环、AI心智、权限落地三个维度综合评估，产品仍处于 Demo 阶段。以下是逐条诊断。

### 一、功能缺失类（代码有骨架、无血肉）

### 二、流程缺失类（断头路，无闭环）

| 编号 | 缺陷 | 代码证据 | 严重度 |
|------|------|---------|--------|
| P-01 | 视频采集流程断裂 | 上传 API (`POST /api/tasks/{id}/upload`) → 存文件到 `uploads/` → 写数据库 `collect_uploads` → **结束**。不触发分析、不回调、不通知 | 🔴 阻塞 |
| P-02 | 任务下发→执行→反馈无回路 | 总部创建任务 → 门店看到 → 门店上传 → **总部不知道上传了什么、解析了什么** | 🔴 阻塞 |
| P-03 | AI对话无上下文持久化 | 对话历史存 `conversations` 表，但 Agent 无记忆：每次对话独立无关联，不会记住"上次我让你分析广安店，现在继续" | 🟡 中等 |
| P-04 | 无调价→执行→追踪闭环 | 如果能生成调价方案，无下发机制、无执行确认、无效果追踪 | 🔴 阻塞 |
| P-05 | 无异常→通知→处理闭环 | 脚本报错、解析失败、超时——只记录 error_message，没有任何通知机制 | 🟡 中等 |

### 三、AI心智缺失类（被动工具，非智能体）

| 编号 | 缺陷 | 代码证据 | 严重度 |
|------|------|---------|--------|
| A-01 | 纯粹被动问答 | `agent_loop.py` 只响应用户输入，不会主动发现异常、主动推送预警、不会说"广安店的可口可乐连续3天比竞品贵，要不要调价？" | 🔴 阻塞 |
| A-02 | 无规划与自主执行能力 | Agent 每次最多5轮工具调用（`max_iterations=5`），无长期目标规划，无法分解复杂任务为多步骤自动执行 | 🔴 阻塞 |
| A-03 | 无记忆与学习能力 | 每次对话独立，不积累"用户偏好/门店特征/历史决策"。无长期记忆（`MEMORY.md` 尚未整合进 Agent loop） | 🟡 中等 |
| A-04 | 工具调用浅层 | 仅5个注册工具（`get_time/list_stores/list_competitors/run_script/search_web`），无数据库查询工具、无数据分析工具、无决策生成工具 | 🟡 中等 |
| A-05 | 无多轮推理链 | LLM返回工具调用→执行→返回结果→LLM汇总→结束。无"分析→发现问题→深入查询→再分析→出方案"的推理链 | 🟡 中等 |

## 模块2：四阶全周期进化落地总路线

```
阶段0（当前）        阶段1（2.5天）        阶段2（15-45天）        阶段3（45天+）
看得见·Demo          可用·流程闭环         好用·主动赋能          商用·自主决策
──────────────────────────────────────────────────────────────────────────────
被动问答             全流程自动化          AI主动服务            虚拟店长自主经营
演示能力             零人工兜底            降80%人工重复          成熟智能体
```

### 阶段0：现状（看得见·纯展示Demo）
- **定位**：有基础骨架，可以演示对话和基础功能
- **核心价值**：验证了多租户架构、任务采集系统、Agent Loop 技术可行性
- **核心缺陷**：无闭环、无差异化、无主动能力、6/10模块未落地
- **验收标准**：当前状态即基线，不做验收

### 阶段1：短期攻坚（可用·流程闭环 / 2.5天）
- **定位**：打通核心链路，消灭所有断头路，实现可上线使用的MVP
- **核心价值**：门店可独立使用，总部可管控全局，数据采集有始有终
- **整体落地目标**：①前端彻底分家 ②视频采集AI解析闭环 ③运营诊断可运行 ④定价算法上线 ⑤人工复核机制
- **验收标准**：三角色分别登录 → 看到不同界面 → 总部下发任务 → 门店接收上传 → AI自动解析 → 总部复核入库 → 定价方案输出（全链路可跑通）

### 阶段2：中期迭代（好用·主动赋能 / 15-45天）
- **定位**：AI从被动变为主动，从问答工具变为经营参谋
- **核心价值**：AI主动监控数据异常、主动推送预警、主动给建议，减少80%人工盯数据的工作
- **整体落地目标**：①微信小程序上线 ②AI主动巡检+异常推送 ③选品规划+商品分层 ④数据看板可视化 ⑤用户偏好学习
- **验收标准**：门店店主每天打开小程序 → 看到AI推送的"今日异常提醒" → 一键查看诊断报告 → 无异常则"一切正常"

### 阶段3：长期成熟（商用·自主决策 / 45天+）
- **定位**：AI从参谋变为准店长，具备自主经营决策能力
- **核心价值**：AI自动发现经营问题→自主制定方案→总部审批→自动执行→效果追踪→自动优化，形成完全自主的经营决策闭环
- **整体落地目标**：①自主经营决策引擎 ②供应链接管 ③财务自动核算 ④美团数据自动接入 ⑤多门店协同调度
- **验收标准**：AI在没有人工干预的情况下，自主完成一次从发现问题→制定方案→执行→追踪的完整经营闭环

#### 3.1.3 本阶段角色能力落地标准

**总部（admin/manager）新增能力：**
- 登录后看到完整10模块界面 + 全部数据资产（13,982条）+ 可切换任意门店
- 能创建采集任务并指定门店+商品清单+截止时间
- 能看到所有门店的采集上传记录和AI解析结果
- 能对低置信度解析结果进行人工复核修正
- 能查看任意门店的比价矩阵和定价方案
- 能管理用户（admin专属）

**门店（store_owner）新增能力：**
- 登录后看到7模块界面（选址/建店/品牌不可见）+ 仅本店数据资产
- 只能看到本店被分配的任务
- 只能上传本店任务的采集视频（有前端格式/大小校验）
- 能看到本店上传的解析状态和结果（失败可重传）
- 能查看本店的定价建议
- 门店选择器固定本店，不可切换

**权限边界明确：**
- store_owner：不可见竞品全局数据、不可见其他门店数据、不可见系统状态、不可创建任务、不可查阅用户列表
- manager：可见全部数据但不可改系统配置、不可管用户
- admin：全权限

### 4.1 视频采集AI解析全闭环落地路线

**现状**：上传→存文件→写数据库→结束。video_pipeline.py（847行RapidOCR+帧分析）从未被调用。

**目标**：上传→校验→异步解析→入库→复核→修正→最终入库→溯源

**落地步骤（阶段1·Day 2-3）：**

```
步骤1：前端上传校验（Day2 D2-1）
├─ 格式：仅MP4/MOV，其他格式弹窗拦截
├─ 大小：≤200MB，超限前端FileReader预检+拦截
├─ 时长：读取video.duration，10s-10min，超出弹窗
├─ MD5：CryptoJS计算hash，先调API查重复→已存在则提示"已上传过"
└─ UI：上传进度条 + 校验结果实时反馈

步骤2：后端上传强化（Day2 D2-2）
├─ 任务归属：task.store_id必须∈user_stores
├─ 重复拦截：SELECT COUNT(*) FROM uploads WHERE task_item_id=? AND file_path包含同MD5
├─ 超时拦截：task.deadline < now() → "任务已截止"
└─ 状态联动：上传成功 → task_items.status='done' + task.status='in_progress'

步骤3：AI异步解析（Day2 D2-3）
├─ 解析流程适配 video_pipeline.py：
│   ├─ 抽帧（每秒1帧，video_pipeline已有FPS_INTERVAL=1）
│   ├─ RapidOCR识别（已有，HAS_OCR标识）
│   ├─ LLM语义聚合（已有）
│   └─ 输出：[{product_name, price, spec, promotion_text, confidence}]
├─ 调用方式：
│   ├─ 上传接口立即返回（不等待解析）
│   ├─ 后台 subprocess.run python3 scripts/video_pipeline.py <file_path> --task_id=X
│   ├─ 解析完成回调：读pipeline输出JSON → 写price_data → 更新upload状态
│   └─ 并发控制：同时最多3个解析任务（asyncio.Semaphore）
└─ 失败重试：最多2次，指数退避(30s/120s)

步骤4：解析失败退回（Day2 D2-4）
├─ upload.status: pending → processing → success/failed
├─ failed时写error_message（如"未检测到价格标签""画面模糊"等）
├─ 门店端轮询 GET /api/tasks/{id}/uploads → 显示状态图标🔄✅❌
└─ 失败记录旁显示"重新上传"按钮

步骤5：人工复核（Day3 D3-1）
├─ 新增表 price_corrections(id, price_data_id, field, original_value, corrected_value, corrected_by, corrected_at)
├─ 总部界面：低置信度(<0.7)记录列表 → 逐条审核 → 修正 → 写留痕
├─ 修正后price_data.confidence=1.0, is_verified=1
└─ 可按修正人/时间追溯

最终入库：price_data(is_verified=1) → 成为比价矩阵和定价算法的可信数据源
```

## 模块6：整体项目风险点 & 可行性评估

### 6.1 风险矩阵

| 风险 | 阶段 | 严重度 | 可能性 | 详情 | 缓解措施 |
|------|------|--------|--------|------|---------|
| Qwen视觉模型精度不达标 | 阶段1 | 🔴高 | 中 | Qwen对便利店铺内视频中密集小字价签的OCR准确率可能<70%，导致AI解析confidence过低，大量记录需人工复核 | ①先用10段真实视频测试Qwen准确率②accuracy<60%则改用RapidOCR(video_pipeline已有)+LLM语义修正的混合方案③设定人工复核的最低置信度阈值 |
| 微信小程序审核不通过 | 阶段2 | 🟡中 | 中 | 类目选择、服务器配置、内容审核可能不通过 | ①提前2周提交审核②准备审核说明文档③提供测试账号 |
| 网络延迟导致解析超时 | 阶段1 | 🟡中 | 低 | 大视频文件（200MB）上传到阿里云ECS→Python脚本调用Qwen→整体链路可能超时 | ①视频大小限制200MB合理②异步解析不阻塞上传响应③后台处理时间无用户感知 |
| 门店端使用者不配合 | 阶段2 | 🟡中 | 中 | 店员可能不会使用小程序、忘记上传、上传不规范视频 | ①设计极简操作流（3步完成）②上传前明确指引（示范视频）③总部端可催办 |
| 数据量不足导致AI分析无力 | 阶段2 | 🟡中 | 中 | 当前数据全部来自静态Excel，门店日常经营数据（销售/库存/客流）实时流入渠道缺失 | ①阶段1优先把视频采集闭环跑通（这是增量数据来源）②阶段2接入POS/收银系统导出的Excel③与门店达成每日上传销售数据的SOP |
| aliyun ECS性能瓶颈 | 全阶段 | 🟢低 | 低 | 1核2G的ECS同时跑FastAPI+Qwen视频解析+定时任务可能吃力 | ①当前并发量极小（2门店）足够②可随时升级到2核4G③视频解析可改为异步队列 |
| DeepSeek API不稳定 | 全阶段 | 🟢低 | 低 | agent_loop已有fallback机制（volcengine→deepseek），但两个都可能不稳定 | ①已有重试降级②增加第三个备用API（智谱GLM） |

### 6.2 可行性判断

| 阶段 | 可行性 | 依据 |
|------|--------|------|
| 阶段1（2.5天） | ✅ **完全可行** | 全部改动在现有代码架构内（index.html改前端渲染、agent_loop.py修bug+加权限注入、app.py改API返回逻辑、video_pipeline.py已存在只需对接调用），无新技术栈、无外部依赖、无审批流程阻塞 |
| 阶段2（15-45天） | ✅ **部分可行** | 技术可行（Qwen已部署、FastAPI成熟），但微信小程序资质审核时间不可控（5-30天），建议先行启动 |
| 阶段3（45天+） | ⚠️ **需分步验证** | 整体可行，但自主决策部分需要阶段1+2积累足够的历史决策数据后，才能验证AI决策的准确率 |

### 6.3 迭代节奏优化建议

1. **阶段1绝不放过**：这是从0到1的质变，一次性解决所有断头路
2. **阶段2优先启动小程序资质**：审核周期不可控，越早提交越好
3. **阶段2中AI主动巡检可以先做（不依赖小程序）**：Web端先落地，小程序同步开发
4. **阶段3不要急于自主执行**：先从"AI提案→人审批"开始，积累6个月决策数据后再放开自动执行
