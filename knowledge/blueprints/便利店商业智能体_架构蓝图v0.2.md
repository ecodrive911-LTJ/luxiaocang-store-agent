# 便利店商业智能体 — 架构蓝图 v0.2

> 「不是工具，是一个能独立运行的经营决策Agent。从数据采集到分析决策到任务执行，全链路闭环。」

---

## 一、核心定位（升级）

**一个可独立部署、持续进化的便利店经营决策Agent。**

- **不是**：一个出报告的工具
- **是**：一个能采集数据→分析问题→给出方案→追踪执行的闭环系统

### 与v0.1的关键差异

| 维度 | v0.1 | v0.2 |
|------|------|------|
| 部署形态 | 嵌在主Agent里的Skill | 独立Agent，可独立部署 |
| 数据采集 | 单一来源（高德API） | 多平台交叉验证体系 |
| 运行终端 | PC端 | PC端 + 手机端（未来） |
| 数据流 | 人工导入为主 | 端侧采集 + 云端分析 + 端侧执行 |
| 商品级数据 | 缺失 | 核心能力之一 |

| 层 | 职责 | 技术形态 |

### 3.2 多平台交叉验证体系

**原则：单一数据源不可信，至少两个来源交叉验证。**

#### 环境数据采集源

| 数据类型 | 主要源 | 验证源 | 采集方式 |
|----------|--------|--------|----------|
| POI/便利店分布 | 高德API（关键字搜索） | 百度地图API + 美团/大众点评 | API + 爬虫 |
| 超市/竞品分布 | 高德API | 58同城 + 大众点评 | API + 爬虫 |
| 人口热力 | 百度慧眼 | 高德人口数据 | API |
| 商圈业态 | 美团外卖/闪购 | 饿了么 + 实地采集 | 爬虫 + 端侧 |
| 周边租金 | 58同城/安居客 | 贝壳找房 | 爬虫 |

#### 商品数据采集源（重点）

| 数据类型 | 采集源 | 采集方式 | 优先级 |
|----------|--------|----------|--------|
| **竞品商品价格** | 美团闪购/外卖 | 浏览器自动化/爬虫 | P0 |
| **竞品商品价格** | 饿了么 | 浏览器自动化/爬虫 | P0 |
| **竞品商品价格** | 大众点评 | 爬虫 | P1 |
| **竞品品类结构** | 实地拍照 + OCR | 手机端采集 | P0 |
| **批发价格** | 1688 API | API | P1 |
| **品牌商品信息** | 各品牌官网/电商 | 爬虫 | P2 |
| **促销活动** | 美团/饿了么/海报 | 爬虫 + OCR | P1 |
| **新品上市** | 零食量贩店公号/抖音 | 爬虫 + 监控 | P2 |

#### 交叉验证算法（初步设计）

```
对于每个采集目标（如：某便利店的可口可乐330ml价格）：

1. 从N个来源分别获取数据点 {S1, S2, ..., Sn}
2. 时间戳对齐（同一时间窗口内的数据）
3. 置信度评分：
   - 来源可靠性权重 W_src（官方API > 平台数据 > 爬虫 > 推算）
   - 时间新鲜度权重 W_time（24h内 > 7天内 > 30天内）
   - 数据一致性权重 W_consensus（多源一致 > 单源）
4. 综合置信度 = Σ(W_src × W_time × W_consensus) / N
5. 若综合置信度 ≥ 0.7 → 采信加权平均值
6. 若综合置信度 < 0.7 → 标记"待验证"，触发补充采集
```

### 3.3 商品级数据采集方案（详细）

**这是整个Agent的核心能力之一，必须有具体可行的技术路径。**

#### 方案A：美团闪购/外卖商品价格采集（P0）

```
技术路径：
1. 浏览器自动化（xbrowser）
   - 打开美团闪购/外卖
   - 定位到目标便利店
   - 遍历商品分类页
   - 抓取：商品名、价格、规格、促销价、月销量
   - 输出：结构化商品价格表

2. 数据字段：
   - store_name: 门店名称
   - platform: 平台（美团/饿了么）
   - product_name: 商品名
   - spec: 规格
   - price: 原价
   - promo_price: 促销价
   - monthly_sales: 月销量
   - category: 平台分类
   - crawl_time: 采集时间

3. 频率：每周一次（竞品价格监控）
```

#### 方案B：实地拍照 + OCR（P0，手机端核心功能）

```
技术路径：
1. 手机端拍摄竞品货架/价签
2. 上传到云端
3. OCR识别：
   - 商品名
   - 价格
   - 促销信息
4. 与已有商品库匹配
5. 输出：竞品价格快照

适用场景：
- 竞品门店货架拍照
- 自己门店货架审计
- 供应商送货单拍照录入
- 促销海报采集
```

#### 方案C：1688批发价格监控（P1）

```
技术路径：
1. 1688商品搜索API
2. 按SKU/商品名匹配
3. 抓取批发价格区间
4. 与采购价对比，发现更优供应商

输出：
- 商品名 → 批发价格区间
- 供应商排名
- 价格趋势
```

#### 方案D：零食量贩店商品监控（P1）

```
目标：好想来、赵一鸣、零食很忙等量贩零食店
技术路径：
1. 微信小程序/公众号商品页
2. 抖音团购商品页
3. 美团/饿了么店铺页
4. 定期抓取新品、价格变动

价值：
- 量贩零食店是便利店直接竞品
- 它们的选品趋势直接影响便利店选品
```

## 六、闭环机制

### 6.1 数据闭环

```
采集 → 清洗 → 分析 → 决策 → 执行 → 反馈 → 再采集
  ↑                                              │
  └──────────────────────────────────────────────┘
```

### 6.2 知识闭环

```
经验 → 沉淀为知识库 → 指导下次决策 → 验证效果 → 更新知识库
```

### 6.3 具体闭环场景

| 场景 | 闭环路径 |
|------|----------|
| 竞品降价 | 采集到价格变化 → 分析影响 → 生成调价建议 → 手机端推送 → 执行后追踪销售 |
| 商品缺货 | 手机端报告缺货 → 分析替代方案 → 推荐替代品+供应商 → 下单 → 验证到货 |
| 新品发现 | 采集到竞品新品 → 分析市场潜力 → 建议是否跟进 → 采购 → 追踪销售 |
| 滞销预警 | 销售数据连续下滑 → 诊断原因 → 建议清仓/换陈列/调价 → 执行 → 追踪效果 |
| 季节切换 | 日历触发 → 分析历史数据 → 生成季节选品方案 → 采购 → 追踪销售 |

## 八、技术选型（初步）

| 组件 | 选型 | 理由 |
|------|------|------|
| Agent框架 | OpenClaw | 已有基础，支持独立workspace |
| LLM | GLM-5.2 / 可切换 | 中文能力强，成本低 |
| 地图API | 高德(关键字) + 百度 | 交叉验证 |
| 爬虫 | xbrowser + requests | 浏览器自动化 + 轻量爬虫 |
| OCR | RapidOCR | 已验证可用 |
| 数据存储 | Excel + JSON | 轻量，未来可升级SQLite |
| 定时任务 | OpenClaw cron | 已有能力 |
| 手机端 | 待定（小程序/Flutter/RN） | Phase 5再定 |
